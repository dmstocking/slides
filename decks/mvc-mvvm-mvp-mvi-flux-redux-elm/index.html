Elm (Flux)

  • Functional
  • Unidirectional
  • Declarative
  • Data Seperation

Model


type alias Model = Int

init : Model
init =
  0
  • Plain old Elm object

View


view : Model -> Html Msg
view model =
  div []
    [ button [ onClick Decrement ] [ text "-" ]
    , div [] [ text (String.fromInt model) ]
    , button [ onClick Increment ] [ text "+" ]
    ]
  • The view is just a function
  • UI elements are just data
  • Pure function

Update


type Msg
  = Increment
  | Decrement

update : Msg -> Model -> Model
update msg model =
  case msg of
    Increment ->
      model + 1

    Decrement ->
      model - 1
  • Update is just a reducer
  • Pure function

Elm (Flux) Takeaways

  • Everything is a function
  • State is self contained and isolated
  • You cannot get anouther views data

Redux

  • Not really an architecture
  • Central state management
  • Elm architecture for your whole app
  • All components have access to entire app state

Redux Takeaways

  • You don't need complicated ways to pass state around
  • Business requirements don't care how "clean" your code is
